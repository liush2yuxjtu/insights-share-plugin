PM Script

这段演示真正要讲的是：一个陌生用户，也能在你的 Mac 上把整套流程亲手跑通。

旧录像的问题，不是链路不通，而是视角不对。它录的是系统怎么自证；新的整屏录像录的是用户如何从冷启动开始，先拉起服务，再完成 Demo、CLI、Validation，并在最后回到 handout 看到自动回填的通过状态。

打开现场验收页 打开 Dashboard 打开最新验证报告

一句话价值主张

它把“AI agent 会自己跑”这件事，翻译成 PM 和评审也能看懂的用户证据：服务能拉起、页面是活的、输入真的打进会话、结果最终能被验证。

为什么旧录像不够

如果录像从 daemon 已启动之后才开始，评审会自然怀疑：是不是后台早就准备好了？新的录像开场先给终端，把“这是现场冷启动”这件事讲清楚，再进入浏览器操作。

三条最强证据

  • Dashboard 会新增现场刚触发的 session。
  • CLI 页发送的固定 prompt 会出现在 tmux pane。
  • Validation 会收敛成 `6/6 通过` 的闭环结果。

用户完整流程分镜

  1. 终端执行启动命令，证明服务是现场拉起的。
  2. 进入 `/handout`，先理解这场验收为什么可信。
  3. 打开 Dashboard 点击 Demo，看新增 session。
  4. 进入 CLI 发送固定提示词，看网页真实驱动 tmux pane。
  5. 回到 Dashboard 跑 Validation，再回 handout 看 Step 3 / 4 / 5 自动变绿。

如何现场讲这段视频

第一句先讲“这不是预录好的后台状态，我现在先把服务拉起来”；第二句讲“Dashboard 上新增的 session 都是我刚点出来的”；第三句讲“CLI 那条中文 prompt 真的进了 tmux”；最后用 Validation 收尾，告诉评审这不是只会跑 demo，而是能被验收。
